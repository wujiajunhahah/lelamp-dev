# LeLamp Dashboard 中文化与手机优先改版设计

Date: 2026-04-14
Status: Approved for implementation
Scope: `lelamp/dashboard/web/` 本地仪表盘的中文展示层、层级重组、手机端适配

## 1. 背景

现有 LeLamp dashboard 已经满足本地控制、状态轮询和诚实降级，但界面气质偏英文工程控制台，不适合作为演示时直接给围观者或手机端访客查看。

本次改版不改 API，不改控制链路，不改轮询模型。重点只做展示层升级：

- 全量中文化，语气从工程面板转为展台演示感
- 首屏先表达“灯现在是什么状态”，不是先展示技术字段
- 手机端优先显示状态，再显示动作，最后显示现场信息
- 保留诊断能力，但把技术细节下沉，避免抢走第一视觉层级

## 2. 用户偏好结论

本次设计以用户在脑暴阶段确认的偏好为准：

- 视觉方向：偏展台演示感，温暖、有点拟人、适合给别人看
- 手机端优先级：先看当前状态和气质
- 主要观众：围观演示的人，而不是只有操作者自己

## 3. 设计目标

- 让第一次打开页面的人在 3 秒内知道灯是否“醒着 / 休息中 / 忙碌 / 异常”。
- 保留现有动作入口，但把文案改成更自然的中文指令。
- 保持“Pi 单机可用、未接台灯也能诚实显示错误或未知”的能力。
- 保证 Raspberry Pi 屏幕和手机浏览器都能稳定显示，不依赖新的前端构建工具。

## 4. 非目标

- 不新增新的后端接口或 WebSocket。
- 不修改动作执行语义、灯光默认值、录制文件格式。
- 不把诊断信息做成独立页面。
- 不在这次改版中引入摄像头、语音、蓝牙等新控制入口。

## 5. 信息架构

页面按“双层展台控制台”组织，具体是三层结构：

### 5.1 第一层：当前状态

首屏 hero 区负责表达灯的当前状态与整体气质：

- 大号中文状态词，如“醒着 / 休息中 / 忙碌 / 注意一下”
- 一句结果描述，解释当前状态意味着什么
- 4 个简化状态卡：灯光、声音、电机、现场连接
- 顶部保留连接、系统状态、当前动作等实时 pill，但视觉权重低于 hero

### 5.2 第二层：动作入口

控制区保留现有能力，但重命名为更适合演示的中文：

- `Startup` → `启动灯`
- `Return Home` → `回到待机`
- `Play Motion` → `播放动作`
- `Shutdown Pose` → `进入休息`
- `Warm Amber` → `暖黄灯光`
- `Light Off` → `关闭灯光`

录制列表仍然存在，但作为“动作选择”的一部分展示，而不是裸技术字段。

### 5.3 第三层：现场信息

把技术与诊断信息统一放在下层面板，文案变成“现场信息 / 设备详情”，而不是直接叫 Diagnostics：

- 可访问地址
- 动作库
- 硬件提示
- 连接建议
- 配置片段
- 错误信息

其中错误信息只在有问题时明显高亮；正常时保持安静。

## 6. 文案策略

### 6.1 状态词映射

内部状态仍由运行时提供，但前端需要做中文展示映射：

- `ready`, `solid`, `muted` → `就绪`
- `running`, `transition` → `进行中`
- `warning` → `注意`
- `error`, `offline` → `异常`
- `unknown` → `未知`

运动状态额外做面向演示的翻译：

- `idle` → `醒着`
- `running` → `动作中`
- `homing` → `回位中`
- `error` → `需要检查`
- `unknown` → `未连接`

### 6.2 说明文案

文案要“像在解释一件作品正在干什么”，而不是“解释一个接口字段”：

- 优先使用完整短句
- 保留技术值时，用次级文字显示
- 录制名、URL、配置键名保持原值，不做伪翻译

## 7. 视觉方向

沿用当前深色底 + 暖色高光的方向，但更接近参考项目的“静、稳、展台化”表现：

- 增强 hero 的暖光氛围与层次感
- 降低诊断区的边框对比和视觉占比
- 按钮从“硬控制”改成“软展台指令”，圆角更饱满，排布更整齐
- 手机端把密集网格改为纵向层叠卡片

## 8. 手机端行为

手机端遵循“状态第一屏”：

- 顶部 pills 改成 2 列或单列堆叠
- hero 与状态卡片变成纵向单列
- 动作区按钮变成一列或两列的大触控区域
- 现场信息下沉到页面后半段，不影响首屏观感

不新增手风琴 JS；仅通过视觉层级和内容排序实现“先看状态，后看细节”。

## 9. 技术实现约束

- 保持 `index.html` 中现有关键节点 id，避免破坏现有 JS 与测试
- 允许新增包装节点和展示节点，但不移除现有 API 依赖点
- `dashboard.js` 增加纯前端文案映射与渲染辅助函数，不改 API 合约
- `dashboard.css` 继续使用纯静态 CSS，不引入构建步骤

## 10. 验证方案

- 更新 `lelamp/test/test_dashboard_web.py`
- 先写失败测试，覆盖：
  - 中文主标题与核心区域文案
  - 手机端响应式断点下的新布局契约
  - JS 初次渲染时的中文按钮和状态映射
- 再修改 HTML/CSS/JS 让测试通过
- 最后跑 dashboard 相关测试，确认无 API 回归
